# Issues and Project Management Plan

In order to effectively track work and measure progress, this repository will leverage the robust capabilities of GitHub Projects and Issues. This strategy covers everything from broad organization down to granular issue tracking.

## Core Elements

### 1. GitHub Projects and Insights
[GitHub Projects](https://docs.github.com/en/issues/planning-and-tracking-with-projects) serves as our central hub for adaptability and work tracking.
- **Insights**: We utilize customizable charts built from project data to quickly review progress and spot bottlenecks at a glance.
- **Project Items & Fields**: Through organization-level issue fields, structured metadata is continuously captured, providing visibility on `priority`, `effort`, and `target dates`.

### 2. Issues, Sub-issues, and Progress
[GitHub Issues](https://docs.github.com/en/issues/tracking-your-work-with-issues/about-issues) are used universally for tracking bugs, tasks, feedback, and ideas.
- **Hierarchy Fields**: We actively use **parent issue** and **sub-issue** fields. Large tasks or epics are broken down, making progress highly visible as sub-tasks move through completion states.
- **Pull Request Links**: Using **pull request fields**, reviewers and PR statuses are surfaced directly on the board for a transparent workflow from branch creation to merge.

### 3. Sprints and Broad Planning
- **Iterations**: For short-term granular work, [Iteration fields](https://docs.github.com/en/issues/planning-and-tracking-with-projects/learning-about-projects/about-iterations) are used to group items and sprint effectively.
- **Milestones**: For sweeping features or versions (like `.NET 10 migration`), [Milestones](https://docs.github.com/en/issues/using-labels-and-milestones-to-track-work/about-milestones) are employed to track broad release progress across associated issues and pull requests.
