End-to-End ETL Architectural Report

Bridging the Relational-to-Hierarchical Impedance Mismatch in SqlHydra

The Challenge: Impedance Mismatch

Moving data from an upstream Relational Database (Oracle) to a downstream Document/Search Engine (Elasticsearch) introduces a massive structural conflict. Oracle thrives on strict normal forms (1NF, 2NF, 3NF) requiring multiple tables and foreign keys. Elasticsearch penalizes JOIN operations heavily, preferring highly denormalized, nested JSON documents.

Our ETL architecture solves this by leveraging FSharp.Data for loss-less type extraction and DuckDB as an analytical transformation engine.

The Pipeline Architecture

📦
1. Upstream Extraction (Oracle)
Raw, highly normalized relational data is queried from Oracle and safely extracted into TSV format.
🛡️
2. Type-Safe Staging (FSharp.Data)
The TSV is ingested via FSharp.Data.CsvProvider. This guarantees that numbers, decimals, and datetimes are not silently corrupted into strings during transit.
🦆
3. Transformation (DuckDB + Parquet)
Data is flushed into highly compressed Parquet files. DuckDB is then booted up to perform the heavy lifting of the ETL: aggregating multiple relational rows (e.g., Order + OrderLines) into hierarchical JSON arrays via high-performance analytical SQL.
🔍
4. Downstream Ingestion (Elasticsearch)
The transformed, hierarchical JSON is indexed into Elasticsearch using the Bulk API, ready for lightning-fast search.

Elasticsearch Data Modeling Litmus Test

To prove our SqlHydra.Elasticsearch AST (esquery) can handle any structural paradigm, we tested three distinct ES mapping strategies against our ETL payload:

Strategy A: Flattened

Model: A single, wide table. Denormalizes Order and OrderLines entirely.

Pros: Extremely fast to query. Trivial to ETL.

Cons: Heavy data duplication (Order details repeated per line).

AST Support: Standard RMatch and RTerm.

Strategy B: Nested Array

Model: An Order document containing a JSON Array of OrderLine objects (nested type).

Pros: Perfect balance. Keeps data grouped hierarchically and enables complex cross-field querying.

Cons: Expensive to update a single line without reindexing the parent.

AST Support: Verified using the nested "Path" (Query) operation.

Strategy C: Parent/Child Join

Model: Independent Order and OrderLine documents joined by a routing join datatype.

Pros: Closest to the Oracle relational model. Allows updating child lines independently.

Cons: High memory overhead; queries are significantly slower.

AST Support: Verified using the hasChild "type" (Query) operation.

Conclusion

The Litmus Test was a complete success. The tests verified that our F# to Elasticsearch compiler (esquery { }) can fluidly target any of the 3 modeling strategies. DuckDB proved to be the perfect bridge engine to absorb the relational complexity and output structured JSON ready for NoSQL ingestion.
