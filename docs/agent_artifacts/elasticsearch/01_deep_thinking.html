Deep Thinking: ES Architecture

Architecting Elasticsearch 8/9 Integration for SqlHydra

The F# Quotation Challenge

SqlHydra translates F# Quotations directly into SQL strings for relational databases. Elasticsearch is fundamentally a NoSQL document store operating on JSON Query DSL. Mapping strict relational logic (JOINs, complex aggregates) to ES requires a paradigm shift.

ES SQL vs. ES|QL

Elasticsearch 8+ introduces ES|QL (Elasticsearch Query Language) alongside the existing ES SQL API. ES SQL allows standard `SELECT` syntax, which aligns perfectly with SqlHydra's current generation. ES|QL uses piped commands (`from... | where... | keep...`).

The "Good Parts" Extrapolation

SqlHydra's strong suit is type-safe query generation and schema extraction. We can leverage the `information_schema` (or `_mapping` API in ES) to generate F# records for ES indexes. Query contexts will then translate standard LINQ/CE directly into ES SQL REST calls.

Phase 1: Schema Provider (Mappings -> F#)

Read the ES `_mapping` API to extract index fields. Translate ES types (text, keyword, integer, boolean) into F# primitives. Generate `ISqlHydraDbProvider` for ES.

Use `Elastic.Clients.Elasticsearch` to pull index definitions. Map nested objects as complex records or JSON strings, maintaining SqlHydra's philosophy of strong typing.

Phase 2: Translation Engine (Quotations -> ES SQL)

Instead of building heavy JSON DSL, compile F# AST into standard SQL strings and POST them to the ES `_sql` endpoint.

The `/_sql?format=json` endpoint accepts standard SQL. This allows us to reuse 80% of SqlHydra's existing SQL AST builder! We just need a dialect specific to ES SQL limitations.
