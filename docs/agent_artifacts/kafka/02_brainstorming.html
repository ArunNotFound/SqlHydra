Brainstorming: Kafka Ecosystem

Confluent.Kafka & FsKafka

Ecosystem Wrappers
Confluent.Kafka is the standard .NET client, wrapping `librdkafka`. FsKafka offers a more idiomatic F# wrapper.

Both are low-level producer/consumer APIs. SqlHydra's role wouldn't be to wrap these, but to feed them perfectly typed F# records generated from the remote Schema Registry. Idea: SqlHydra.Kafka.Provider could auto-generate Consumer/Producer loops with strict typing.

ksqlDB Integration

Streaming SQL
ksqlDB allows querying Kafka streams using SQL. Since SqlHydra generates SQL, we can construct push queries (`SELECT EMIT CHANGES`) that translate into continuous async streams in F# (`IAsyncEnumerable<'T>`).

Idea: Instead of returning a List, SqlHydra translates `query { ... }` into an ongoing Task stream!

Event Sourcing / CDC

Data Pipeline (Debezium)
If users use Debezium to stream Postgres/SQL Server changes into Kafka, SqlHydra already has the relational schema! We can generate strongly-typed "Change Data Capture" event structures.

Idea: Map relational `Customer` table to a Kafka `Customer_CDC_Event` F# record seamlessly.

Schema Registry Generation

Avro / Protobuf
SqlHydra accesses the Confluent Schema Registry API, downloads the Avro/Protobuf schema, and translates the JSON AST into F# records.

Idea: Reuses our learnings from Symphony's Native JSON AST mapping!
